# 测试平台 PRD

## 1. 背景

当前已有测试框架可以执行用例，但执行结果分散在各执行机本地，不方便统一查看和统计。

需要搭建一个公司内网测试平台。测试框架在每条用例执行完成后，将该用例的执行结果和报告文件上报到平台，平台负责保存报告文件、查询和统计。

## 2. 核心约束

- 测试框架负责执行用例，平台不负责调度和执行。
- 每条用例执行完成后独立上报，不需要测试套件或测试任务维度。
- 每条用例会单独生成测试报告，上报时必须携带报告文件，由平台保存。
- 不关注台架维度，平台通过执行机 owner 区分不同环境，owner 表示个人。
- 测试框架中每个用例都有唯一且稳定的用例 ID。
- 首版只做内网使用。

## 3. 功能需求

### 3.1 测试结果上报

测试框架每执行完成一条用例后，通过 API 向平台上报一条用例执行结果。

上报内容包含：

- 执行机信息。
- 执行机 owner。
- 用例 ID、名称和模块。
- 用例执行的开始时间、结束时间和耗时。
- 用例执行结果。
- 用例报告文件。
- 失败或异常时的错误类型和错误信息。

平台接收到上报后，应保存报告文件和一条用例报告记录。平台不要求同一批用例归属于某个测试任务或测试套件。

### 3.2 执行机信息

平台需要记录执行机信息，用于按环境统计。

执行机字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| runner_id | 是 | 执行机唯一标识 |
| runner_name | 否 | 执行机名称 |
| runner_owner | 是 | 执行机 owner，用于区分环境，取值为个人 |
| ip | 否 | 执行机 IP |

说明：

- 执行机首次上报时，平台可自动创建执行机记录。
- 同一执行机 owner 变化时，以本次上报值作为本条用例报告的统计归属。

### 3.3 用例报告记录

每次上报生成一条用例报告记录。

用例报告字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| case_report_id | 是 | 平台生成的用例报告 ID |
| idempotency_key | 是 | 幂等键，用于避免重复上报产生重复记录 |
| runner_id | 是 | 执行机唯一标识 |
| runner_owner | 是 | 执行机 owner 快照 |
| case_id | 是 | 用例 ID，由测试框架保证唯一且稳定 |
| case_name | 是 | 用例名称 |
| module | 否 | 所属模块 |
| started_at | 是 | 用例开始时间 |
| ended_at | 是 | 用例结束时间 |
| duration_ms | 否 | 用例执行耗时 |
| result | 是 | 用例执行结果 |
| report_file_path | 是 | 平台保存的报告文件相对路径 |
| report_filename | 是 | 原始报告文件名 |
| report_content_type | 是 | 报告文件 MIME 类型 |
| report_size_bytes | 是 | 报告文件大小 |
| error_type | 否 | 错误类型 |
| error_message | 否 | 错误信息 |
| created_at | 是 | 平台接收时间 |

用例结果：

- `passed`：通过。
- `failed`：失败。
- `skipped`：跳过。
- `blocked`：阻塞。
- `error`：执行异常。

### 3.4 首页看板

首页展示整体执行情况：

- 今日用例报告总数。
- 今日用例通过率。
- 今日失败用例数。
- 按 owner 统计用例数、失败数和通过率。
- 按执行机统计最近执行结果和最近上报时间。
- 最近失败用例列表。

### 3.5 用例报告列表

用例报告列表支持查看和筛选历史执行记录。

列表字段：

- 执行机。
- owner。
- 用例 ID。
- 用例名称。
- 模块。
- 执行结果。
- 开始时间。
- 结束时间。
- 耗时。
- 报告访问入口。

筛选条件：

- 时间范围。
- owner。
- 执行机。
- 执行结果。
- 模块。
- 用例 ID 或用例名称。

### 3.6 用例报告详情

用例报告详情展示单条用例执行结果。

需要展示：

- 执行机和 owner。
- 用例基础信息。
- 执行时间、耗时和结果。
- 失败或异常时的错误类型和错误信息。
- 报告访问入口。

### 3.7 失败用例排查

失败用例列表聚焦展示 `failed` 和 `error` 结果。

支持：

- 按时间范围筛选。
- 按 owner 筛选。
- 按执行机筛选。
- 按模块筛选。
- 按用例 ID 或用例名称搜索。
- 打开对应报告。

### 3.8 统计查询

平台需要支持以下统计：

- 按日期统计用例数、失败数和通过率。
- 按 owner 统计用例数、失败数和通过率。
- 按执行机统计用例数、失败数、通过率和最近上报时间。
- 按用例统计失败次数。

统计口径：

- 用例通过率 = 通过用例数 / 已执行用例数。
- 已执行用例数 = `passed` + `failed` + `error`。
- `skipped` 和 `blocked` 不计入通过率分母，但需要单独展示。
- 平台不计算测试任务通过率，因为首版不建模测试任务。

### 3.9 幂等与重复上报

测试框架每条用例上报时应提供 `idempotency_key`。

平台要求：

- 同一个 `idempotency_key` 重复上报时，不重复生成用例报告记录。
- 重复上报应返回已有用例报告 ID。
- 如果上报失败，测试框架可以使用相同 `idempotency_key` 重试。
- `idempotency_key` 的推荐生成方式是一次用例执行的稳定唯一标识，例如 `runner_id + case_id + started_at` 或测试框架自身的用例执行 ID。

### 3.10 上报接口

`POST /api/v1/test-reports`

请求格式：

- `Content-Type: multipart/form-data`
- `payload`：用例执行结果 JSON 字符串。
- `report_file`：必填，单条用例生成的报告文件。

`payload` 结构：

```json
{
  "idempotency_key": "string",
  "runner": {
    "runner_id": "string",
    "runner_name": "string",
    "runner_owner": "string",
    "ip": "string"
  },
  "case": {
    "case_id": "string",
    "case_name": "string",
    "module": "string",
    "started_at": "2026-06-30T10:00:00+08:00",
    "ended_at": "2026-06-30T10:01:00+08:00",
    "duration_ms": 60000,
    "result": "passed",
    "error_type": "string",
    "error_message": "string"
  }
}
```

响应结构：

```json
{
  "case_report_id": "string",
  "report_url": "string",
  "status": "imported",
  "message": "string"
}
```

说明：

- `report_url` 由平台生成，用于访问平台保存的报告文件。
- 测试框架不得在 payload 中传入外部报告地址。

## 4. 验收标准

1. 测试框架每条用例执行完成后，可以通过 API 上报单条用例结果和报告文件。
2. 平台可以按执行机 owner 查询和统计不同环境的执行情况。
3. 平台可以展示用例报告列表和用例报告详情。
4. 平台可以筛选失败用例。
5. 平台可以展示失败用例的错误信息，并打开平台保存的报告。
6. 同一条用例执行结果重复上报时，不产生重复记录。

## 5. 已确认

1. `runner_owner` 表示个人。
2. 测试框架中每个用例都有唯一且稳定的用例 ID。
3. 每条用例执行完成后会单独生成报告文件。
4. 首版不需要独立日志链接和截图链接字段。
5. 首版不建模测试套件或测试任务。
