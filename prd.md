# 测试平台 PRD

## 1. 背景

当前已有测试框架可以执行用例，但执行结果分散在各执行机本地，不方便统一查看和统计。

需要搭建一个公司内网测试平台。测试框架在用例执行完成后，将本次执行结果上报到平台，平台负责保存、查询和统计。

## 2. 核心约束

- 测试框架负责执行用例，平台不负责调度和执行。
- 用例执行完成后再上报，不需要实时过程监控。
- 没有测试套件维度，平台只关注测试任务和测试用例。
- 不关注台架维度，平台通过执行机 owner 区分不同环境，owner 表示个人。
- 测试框架中每个用例都有唯一且稳定的用例 ID。
- 首版不处理附件保存，只保存必要的报告、日志、截图链接。
- 首版只做内网使用。

## 3. 功能需求

### 3.1 测试结果上报

测试框架执行完成后，通过 API 向平台上报一次完整测试结果。

上报内容包含：

- 执行机信息。
- 执行机 owner。
- 本次执行的开始时间、结束时间、耗时。
- 本次执行的汇总结果。
- 用例执行明细。
- 测试报告、日志或截图链接。

平台接收到上报后，应保存一条测试任务记录，并保存对应的用例结果。

### 3.2 执行机信息

平台需要记录执行机信息，用于按环境统计。

执行机字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| runner_id | 是 | 执行机唯一标识 |
| runner_name | 否 | 执行机名称 |
| runner_owner | 是 | 执行机 owner，用于区分环境，取值为个人 |
| ip | 否 | 执行机 IP |

说明：

- 执行机首次上报时，平台可自动创建执行机记录。
- 同一执行机 owner 变化时，以本次上报值作为本次任务的统计归属。

### 3.3 测试任务记录

每次上报生成一条测试任务记录。

任务字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| run_id | 是 | 平台生成的任务 ID |
| runner_id | 是 | 执行机唯一标识 |
| runner_owner | 是 | 执行机 owner |
| started_at | 是 | 开始时间 |
| ended_at | 是 | 结束时间 |
| duration_ms | 否 | 执行耗时 |
| status | 是 | 执行结果 |
| report_url | 否 | 测试报告地址 |

任务状态：

- `passed`：全部通过或达到通过标准。
- `failed`：存在失败用例。
- `error`：测试框架执行异常或上报数据不完整。

### 3.4 用例结果

平台需要保存每条用例的执行结果。

用例字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| case_id | 是 | 用例 ID，由测试框架保证唯一 |
| case_name | 是 | 用例名称 |
| module | 否 | 所属模块 |
| result | 是 | 执行结果 |
| duration_ms | 否 | 执行耗时 |
| error_type | 否 | 错误类型 |
| error_message | 否 | 错误信息 |
| log_url | 否 | 日志地址 |
| screenshot_url | 否 | 截图地址 |

用例结果：

- `passed`：通过。
- `failed`：失败。
- `skipped`：跳过。
- `blocked`：阻塞。
- `error`：执行异常。

### 3.5 首页看板

首页展示整体执行情况：

- 今日任务总数。
- 今日任务通过率。
- 今日用例总数。
- 今日用例通过率。
- 今日失败用例数。
- 按 owner 统计任务数、用例数、失败数、通过率。
- 按执行机统计最近执行结果和最近上报时间。
- 最近失败任务列表。
- 最近失败用例列表。

### 3.6 任务列表

任务列表支持查看和筛选历史执行记录。

列表字段：
- 执行机。
- owner。
- 任务状态。
- 开始时间。
- 结束时间。
- 耗时。
- 用例总数。
- 失败数。
- 通过率。

筛选条件：

- 时间范围。
- owner。
- 执行机。
- 任务状态。

### 3.7 任务详情

任务详情展示一次执行的完整结果。

需要展示：

- 任务基础信息。
- 汇总结果：总数、通过数、失败数、跳过数、通过率。
- 用例结果列表。
- 失败用例的错误信息。
- 日志、截图、报告链接。

用例列表支持：

- 按结果筛选。
- 按模块筛选。
- 按用例 ID 或用例名称搜索。

### 3.8 统计查询

平台需要支持以下统计：

- 按日期统计任务数、用例数和通过率。
- 按 owner 统计任务数、用例数、失败数和通过率。
- 按执行机统计任务数、失败数、通过率和最近上报时间。
- 按用例统计失败次数。

统计口径：

- 用例通过率 = 通过用例数 / 已执行用例数。
- 任务通过率 = 通过任务数 / 已结束任务数。
- 跳过用例不计入失败，但需要单独展示。

### 3.9 幂等与重复上报

测试框架上报时应提供 `idempotency_key`。

平台要求：

- 同一个 `idempotency_key` 重复上报时，不重复生成任务。
- 重复上报应返回已有任务 ID。
- 如果上报失败，测试框架可以使用相同 `idempotency_key` 重试。

### 3.10 上报接口

`POST /api/v1/test-reports`

请求结构：

```json
{
  "idempotency_key": "string",
  "runner": {
    "runner_id": "string",
    "runner_name": "string",
    "runner_owner": "string",
    "ip": "string"
  },
  "run": {
    "started_at": "2026-06-30T10:00:00+08:00",
    "ended_at": "2026-06-30T10:30:00+08:00",
    "duration_ms": 1800000,
    "status": "passed",
    "report_url": "string"
  },
  "summary": {
    "total_count": 100,
    "passed_count": 95,
    "failed_count": 3,
    "skipped_count": 2
  },
  "cases": [
    {
      "case_id": "string",
      "case_name": "string",
      "module": "string",
      "result": "passed",
      "duration_ms": 1200,
      "error_type": "string",
      "error_message": "string",
      "log_url": "string",
      "screenshot_url": "string"
    }
  ]
}
```

响应结构：

```json
{
  "run_id": "string",
  "status": "imported",
  "message": "string"
}
```

## 4. 验收标准

1. 测试框架执行完成后，可以通过 API 上报完整测试结果。
2. 平台可以按执行机 owner 查询和统计不同环境的执行情况。
3. 平台可以展示任务列表、任务详情和用例明细。
4. 平台可以筛选失败任务和失败用例。
5. 平台可以展示失败用例的错误信息和日志链接。
6. 同一次执行重复上报时，不产生重复任务。

## 5. 已确认

1. `runner_owner` 表示个人。
2. 测试框架中每个用例都有唯一且稳定的用例 ID。
3. 首版不处理附件保存，只保存报告、日志、截图等链接。
